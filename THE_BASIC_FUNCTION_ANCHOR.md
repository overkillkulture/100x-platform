## THE ANCHOR ENFORCEMENT PROTOCOL

**For ANY system:**

### STEP 1: DEFINE THE ANCHOR

**Ask:** What is the ONE basic function this serves?

**Write it down:** Maximum 10 words

**Examples:**
- DNS: "Point domain to IP"
- Word processor: "Type and save text"
- Email: "Send and receive messages"
- Calendar: "Track time commitments"
- Browser: "Access web pages"

**This never changes.**

---

### STEP 2: ANCHOR ALL FEATURES

**For every existing or proposed feature:**

```
┌─────────────────────────────────┐
│  FEATURE ANCHOR TEST            │
├─────────────────────────────────┤
│                                 │
│  Does this feature:             │
│                                 │
│  A) Directly serve basic fn? ✓ │
│  B) Support basic fn? → Optional│
│  C) Unrelated to basic fn? ✗   │
│                                 │
└─────────────────────────────────┘
```

**A = Core interface**
**B = Optional/Settings**
**C = Delete or separate product**

---

### STEP 3: INTERFACE HIERARCHY

**Based on anchor test results:**

**Tier 1: BASIC FUNCTION (always visible)**
- Largest buttons
- Fewest clicks
- No distractions
- **This is 80% of interface**

**Tier 2: SUPPORTING (one click away)**
- Settings
- Preferences
- Optional features
- Advanced options
- **Hidden until needed**

**Tier 3: UNRELATED (separate product)**
- Upsells
- Cross-sells
- Marketing
- **Completely separate interface**

---

### STEP 4: REGULAR ANCHOR CHECKS

**Quarterly review:**

1. What's the basic function? (State the anchor)
2. How many clicks to complete it? (Should be ≤3)
3. How many features added since last check?
4. How many passed anchor test? (Should be <20%)
5. What can be deleted? (Usually 50%+)

**If basic function requires >3 clicks → REDESIGN**

## HOW TO IMPLEMENT THE ANCHOR

### FOR EXISTING SYSTEMS:

**Step 1: Identify the anchor**
- What was the original purpose?
- What do 80% of users actually do?
- What's the ONE thing this should do perfectly?

**Step 2: Audit all features**
- List every feature/option
- Test each against anchor
- Categorize: Core / Supporting / Unrelated

**Step 3: Redesign interface**
- Core features: Tier 1 (always visible)
- Supporting: Tier 2 (settings menu)
- Unrelated: Delete or spin off

**Step 4: Enforce going forward**
- New features must pass anchor test
- Quarterly reviews
- User testing on basic function completion time
- **Ruthlessly delete complexity**

---

### FOR NEW SYSTEMS:

**Step 1: Define anchor FIRST**
- Before writing code
- Before designing UI
- Before feature planning
- **This is the constitution**

**Step 2: Design for anchor ONLY**
- Build interface that does ONE thing
- Test with kindergartner and grandma
- If they can't do it → simplify more

**Step 3: Add supporting features SLOWLY**
- One at a time
- Each must pass anchor test
- Each goes in Settings, not main UI
- **Default to NO**

**Step 4: Maintain forever**
- Resist feature creep
- Resist scope creep
- Resist complexity creep
- **The anchor is law**
